## 4. V2 训练全流程

### 4.1 V2 架构原理

```
VoiceConversionWrapper
├── content_extractor_narrow   (AstralQuantizer: Whisper-small + HuBERT → 窄内容)
├── content_extractor_wide     (同上 → 宽内容)
├── style_encoder              (CAMPPlus: 参考音频 → 192维 speaker embedding)
├── CFM  ← ★ 可微调（流匹配扩散去噪器, 337MB）
│   └── cfm_length_regulator
├── AR   ← ★ 可微调（自回归 Transformer-12层, 359MB）
│   └── ar_length_regulator
└── vocoder                    (BigVGAN 22kHz, 冻结)

推理: 源音频 → content_extractor → CFM(style条件) → AR(逐帧生成) → BigVGAN → 输出
```

**CFM** 负责音色映射（"花丸的声音长什么样"）
**AR** 负责发音节奏（"花丸的说话节奏/停顿习惯"）

### 4.2 说话数据切分算法

针对杂谈/直播回放的逆向目标点切分法：

```
1. silencedetect (-35dB, 0.3s) → 只抓 ≥0.3s 的真停顿
2. 文件 ≤30s → 直接保留
3. 文件 >30s:
   ├─ 按 IDEAL_SEG=15s 生成均匀目标切点: 15s, 30s, 45s, ...
   ├─ 每个目标点附近 [target-5, target+10] 搜索停顿 → 选 dur 最大的
   ├─ 扩大窗口 [target-8, target+15] → [target-10, target+20]
   ├─ 段长 ≤30s → 输出到 normal/
   └─ 段长 >30s → 输出到 manual/（带 _OVER_XXs 后缀）
```

**关键参数：**
- 噪音阈值：`-35dB`（UVR 后干声干净）
- 最短静音：`0.3s`（句间换气/思考停）
- 理想段长：`15s`
- 最大段长：`30s`

**密集说话直播回放适配：** `-20dB, 0.2s`

**结果：**
- 22 个说话文件（8.1 小时）→ 1695 段正常（436 分钟）← V2 训练数据
- 巨型直播文件（135min/60min 等）单独用 `recut_dense.py` 处理

### 4.3 V2 训练成果

| 轮次 | 数据 | 模块 | 步数 | run name | 显存 | 速度 |
|------|------|------|------|----------|------|------|
| V2 R1 | 1695 条说话 | CFM only | 6000 | hanamaru_v2_cfm | ~8GB | ~4.5 it/s |
| V2 R2 | 1695 条说话 | CFM + AR | 6000 | hanamaru_v2_cfm_ar | **11.4GB** | ~2.5 it/s |

**CFM checkpoint:** 336MB/个，12 个共 4GB
**AR checkpoint:** 318MB/个，12 个共 3.8GB

### 4.4 V2 关键 Bug 修复

1. **CAMPPlus BatchNorm 崩溃** — `vc_wrapper.py` 的 `forward()` 中 `style_encoder` 需设 `eval()` 模式（batch=1 训练时 BN 需要 ≥2 样本）
2. **CFM 时间戳 squeeze** — `cfm.py` 训练路径 `t.squeeze()` → `t.squeeze(1).squeeze(-1)`（shape 不匹配）
3. **AstralQuantizer 离线模式** — 恢复原始 `local_files_only=False`，网络通时自动下载
4. **HuBERT 缓存损坏** — 删掉重下（`preprocessor_config.json` 缺失）
5. **train_v2.py 自动清 checkpoint** — `max_keep=1` 改为全保留

## 8. FAQ

**Q: 呼吸有电音怎么办？**
A: V1：`--diffusion-steps` 从 4 提高到 20。V2 天生无电音。

**Q: 音色不够像？**
A: V1：增加训练数据/步数，或提高 `--inference-cfg-rate` 到 1.0

**Q: 跑调？**
A: V1 确认 `--f0-condition True`；V2 无法解决，换 V1。

**Q: 能不能用 V2 唱歌？**
A: 不能。V2 无 F0 引导，唱歌必然跑调。

**Q: V1 100 步比 20 步好吗？**
A: RTF 从 0.7→2.0，听感差异极小。20 步是边际最优。

**Q: V2 similarity-cfg-rate 应该设多少？**
A: 微调后设 0.1~0.3（模型权重主导），零样本设 0.7（参考音频主导）。

**Q: CFM 和 AR 分别管什么？**
A: CFM 管音色（"声音像谁"），AR 管节奏/发音模式（"怎么说话"）。

**Q: 显存 11.4GB 跑 CFM+AR 安全吗？**
A: 刚好，无余量。不要开 num-workers，不要同时跑其他 GPU 应用。

**Q: RTF 是什么意思？**
A: Real-Time Factor。RTF=0.14 → 0.14 秒生成 1 秒音频，比实时快 7 倍。

### 三模型定位总览

| | V1 (唱歌) | V2 (说话) | YingMusic (唱歌) |
|---|---|---|---|
| **模型下载** | HF: `seedvc_v1_15000.pth` | 本仓库 `runs/` 目录 | HF: 5 个 `.pth` 文件 |
| **推理参数** | diff=20, f0=True, cfg=0.7 | diff=30, intel=0.7, sim=0.1 | diff=50~100, f0=True, cfg=0.7 |
| **输入** | 唱歌干声 | 说话干声 | 唱歌干声 |
| **输出** | 花丸唱歌 | 花丸说话 | 花丸唱歌 |
| **音色** | 最像，偏刺耳 | 微调后改善 | 柔和自然，选型灵活 |
| **清新感** | ⭐⭐⭐ | — | ⭐⭐⭐⭐⭐ (cosine/exp1) |

> V1 音色最像但偏刺耳。YingMusic 柔和自然，cosine/exp1 清新、v3 准确、60k 厚重——四模型覆盖不同风格。
>
> **YingMusic 训练规律**：spk_embedding 初始化必须用单条干净 CAMPPlus。CFM cos 越高越清新，越低越厚重。清新感与音色准度是 trade-off。
